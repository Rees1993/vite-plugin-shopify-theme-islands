# Invalidate Revive Only On Ownership Changes

## Parent

- [01-registered-tag-default.md](/Users/alexrees/Documents/Lazer/vite-plugin-shopify-theme-islands/.scratch/registered-tag-default/issues/01-registered-tag-default.md)

## What to build

Refine dev-mode invalidation so the virtual `/revive` module reloads only when effective Island ownership changes. Ordinary implementation edits inside an Island file should remain normal HMR; only changes to whether a file is an Island, what Tag it owns after `tagSource` and `resolveTag()`, or whether it is excluded should invalidate `/revive`.

## Acceptance criteria

- [ ] The plugin tracks enough per-file Island ownership metadata to detect ownership changes semantically.
- [ ] Ordinary Island implementation edits do not invalidate the virtual `/revive` module.
- [ ] Changes to Island membership, effective Tag ownership, or explicit exclusion do invalidate `/revive`.
- [ ] Dev-mode tests distinguish semantic ownership changes from ordinary implementation edits.
- [ ] The implementation works for both default `registeredTag` behavior and explicit `tagSource: "filename"` compatibility mode.

## Agent Brief

**Category:** enhancement  
**Summary:** Make dev-mode `/revive` invalidation semantic so it only reloads when effective Island ownership changes.

**Current behavior:**  
The plugin invalidates the virtual `/revive` module based on coarse Island-set changes. With Registered Tag ownership, ordinary implementation edits risk causing unnecessary invalidation if the plugin does not distinguish ownership changes from normal module edits.

**Desired behavior:**  
Dev mode should reload the virtual `/revive` module only when effective Island ownership changes. Ordinary implementation edits inside an Island file should remain normal HMR. Ownership-relevant changes include whether a file is an Island, which Tag it owns after `tagSource` and `resolveTag()`, and whether it is excluded.

**Key interfaces:**  
- Per-file Island ownership metadata — enough to determine Island membership, effective Tag, and explicit exclusion status  
- Dev invalidation handshake for the virtual `/revive` module — should react only to ownership metadata changes  
- Both Tag source modes — semantic invalidation should work for default `registeredTag` and explicit `filename`

**Acceptance criteria:**  
- [ ] The plugin tracks enough per-file Island ownership metadata to detect ownership changes semantically  
- [ ] Ordinary Island implementation edits do not invalidate the virtual `/revive` module  
- [ ] Changes to Island membership, effective Tag ownership, or explicit exclusion do invalidate `/revive`  
- [ ] Dev-mode tests distinguish semantic ownership changes from ordinary implementation edits  
- [ ] The implementation works for both default `registeredTag` behavior and explicit `tagSource: "filename"` compatibility mode

**Out of scope:**  
- Changing runtime activation semantics or Discovery rules  
- Broad performance work unrelated to `/revive` invalidation correctness
