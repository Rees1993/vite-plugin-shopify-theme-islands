# Default Tag Source To Registered Tag

## Parent

- [01-registered-tag-default.md](/Users/alexrees/Documents/Lazer/vite-plugin-shopify-theme-islands/.scratch/registered-tag-default/issues/01-registered-tag-default.md)

## What to build

Make `registeredTag` the default Tag source in v2.0 so an omitted `tagSource` derives each Island's Tag from a statically-readable `customElements.define("...", ...)` call. Preserve current Discovery behavior and keep `resolveTag()` as the final override layer; only the default Tag derivation path should change.

## Acceptance criteria

- [ ] Omitting `tagSource` behaves the same as `tagSource: "registeredTag"`.
- [ ] Compile derives an Island's default Tag from exactly one static `customElements.define("...", ...)` call in `registeredTag` mode.
- [ ] `registeredTag` mode fails at compile time when an Island file has no readable Registered Tag, or more than one readable Registered Tag.
- [ ] `registeredTag` mode accepts non-kebab-case filenames such as `CardDrawer.ts` as long as the Registered Tag is valid.
- [ ] `resolveTag()` still runs in `registeredTag` mode using the Registered Tag as `defaultTag`.
- [ ] Automated tests cover the new default behavior and strict compile-time failures.

## Agent Brief

**Category:** enhancement  
**Summary:** Make `registeredTag` the default Tag source so Island ownership follows the statically-readable `customElements.define(...)` Tag when `tagSource` is omitted.

**Current behavior:**  
Island Tag ownership defaults to the filename-derived Tag. Compile only reads `customElements.define(...)` as a diagnostic signal and warns when it disagrees with filename ownership.

**Desired behavior:**  
When `tagSource` is omitted, Compile should derive each Island's default Tag from exactly one statically-readable `customElements.define("...", ...)` call. Discovery should remain unchanged. `resolveTag()` should still run afterward and receive the Registered Tag as `defaultTag`. In this mode, non-kebab-case filenames are acceptable as long as the Registered Tag is valid. Compile should fail when an Island file has no statically-readable Registered Tag or has more than one.

**Key interfaces:**  
- Plugin config shape — `tagSource` should accept `registeredTag` and `filename`, with omitted `tagSource` behaving as `registeredTag`  
- Compile-time Tag derivation — the default Tag source should branch on `tagSource` without changing Discovery  
- `resolveTag({ filePath, defaultTag })` — should continue to work in both modes, with `defaultTag` sourced from the selected Tag source  
- Duplicate Tag ownership checks — should remain strict after Registered Tag derivation

**Acceptance criteria:**  
- [ ] Omitting `tagSource` behaves the same as `tagSource: "registeredTag"`  
- [ ] Compile derives an Island's default Tag from exactly one static `customElements.define("...", ...)` call in `registeredTag` mode  
- [ ] `registeredTag` mode fails at compile time when an Island file has no readable Registered Tag, or more than one readable Registered Tag  
- [ ] `registeredTag` mode accepts non-kebab-case filenames such as `CardDrawer.ts` as long as the Registered Tag is valid  
- [ ] `resolveTag()` still runs in `registeredTag` mode using the Registered Tag as `defaultTag`  
- [ ] Automated tests cover the new default behavior and strict compile-time failures

**Out of scope:**  
- Expanding support to `window.customElements.define(...)`, `globalThis.customElements.define(...)`, helper wrappers, computed Tag strings, or scoped registries  
- Changing Discovery rules or mixin-vs-directory Island detection
